All capabilities v3.9.0

This release note lists the changes to all capabilities in this release.

New features

This release provides the following new features:

  • Support for OpenAPI

    Widgets from all capabilities in this collection now support OpenAPI specifications. This change applies to newly generated web clients.

Deprecations

This release includes the following deprecations:

  • In the version released in January 2022:

    • Support ceases for Microsoft Internet Explorer 11

      Note
      		 This date was previously set to January 2021, but support for Microsoft Internet Explorer 11 will now continue to January 2022.

  • In v3.10.0:

    • Support ceases for the use of non-prefixed ApiModule, BASE_PATH, and ConfigurationParameters in the data-ang project. Check the public_api.ts file in data-ang project for the correct prefix to use.

      The new patterns are already supported in v3.9.0 of the data-ang project.

To update an existing deployment to accommodate these deprecations, see Migration.

Breaking changes

This release includes the following breaking changes:

  • All web widgets have been upgraded to version 3.9.0 of the data-ang project, which supports OpenAPI.

This release note explains changes to this capability.

This release provides the following new features:

Batches v3.9.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: Upload and manage ACH Debit Balanced Batch (NACHA)

    Added an OOTB configured implementation of the ACH Debit Balanced Batch (NACHA) variant. This batch type can now be uploaded, managed and is integrated with all the current batch related functionality.

  • Web: ‘Manage batch payments’ and ‘View batch details’ enhancements for supporting both credit and debit batch types

    In order to support credit and debit batches (e.g. ACH batch types), there was a need for the UI to be enhanced. For this the following was done:

    • Both the ‘Manage batch payments’ and ‘View batch details’ widget have separate credit and debit columns, where corresponding credit/debit total batch amounts/number of batch transactions are displayed.

    • ‘Manage batch payments’ widget: When retrieving the list of batches from the BackEnd, an additional attribute has been introduced within the response ‘creditDebitMixedIndicator’. These can contain the following values; Credit, Debit or Mixed (see DBS release notes for details on this). Based on these values either the credit, debit or both credit and debit column will be displayed on the list of batches.

    • ‘View batch details’ widget: When retrieving the batch details from the BackEnd, an additional attribute on batchorder level has been introduced within the response ‘creditDebitMixedIndicator’. This depicts if the batch is a credit, debit or mixed batch type. Based on this value, either the credit, debit or both credit and debit column will be displayed within the batch detail overview.

Notifications v3.9.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: A human-readable timestamp

    A human-readable timestamp was added as an option to the Notification badge widget. This feature can help identify the needed notification among multiple notifications throughout the day.

  • Web: Show more

    The Show more feature was added as an option to the Notification badge and View sticky and popup notifications widgets. This change allows the bank customer to read the full text of the notification if the notification navigates to an external widget. It also solves the issue with the customer not being able to read the long text of a notification in a pop-up when the pop-up timeout is exceeded.

  • Web: Accessibility improvements

    The date of the notification is now readable for screen readers.

Bug fixes

This release includes the following bug fixes:

  • MAINT-10241: Retail Banking Web Apps 2020.08-beta - Notifications take focus causes disruption for users

    The solution was added which allows reading of the notification without changing of the focus. This feature helps visually impaired app users to avoid distractions from their current tasks by new pop-up notification, and at the same time to be informed about new notifications.

Deprecations

This release includes the following deprecations:

  • In v3.10.0:

    • Support ceases for the isTextTruncated method of the bbNotificationsListItemCustomizable slot in the Notification badge widget.

Breaking changes

This release includes the following breaking changes:

  • The old navigation approach for widgets is replaced with the new navigation approach. See Configure communication between widgets.

Payments v3.9.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: Transaction signing in create payment flow

    On omni payment widget and payment wizard widget, it is possible for user to confirm a submitted payment by a chosen transaction signing method.

    • User is asked to confirm via a new dialogue with the confirmation challenge and other payment details.

    • User can enter this confirmation on the same device (in-band) or another device (out-of-band) as per user’s choice.

    • User can see a message for successful confirmation as well as sub-sequent successful submission of payment.

  • Web: Limit Breach report in create payment flow

    Now it is possible for user to see limit breach report after submitting a payment in omni payment widget, which is already provided by endpoint POST https://ips_host:ips_port/gateway/api/payment-order-presentation-spec/client-api/v2/payment-orders. Report includes details of submitted payment, Type of limits breached (Transactional, Daily, Monthly, Quarterly, Yearly) and Threshold, Consumed amounts for each type.

  • Web: Submit and Approve a payment together

    On the review screen in the omni payment and payment wizard widgets, user can choose to 'Submit and Approve' only if the user has the required approval permissions as per the entitlement and workflow setup.

  • Web: Template features in the payment wizard widget

    • Payment wizard is enhanced with ability to save a submitted payment as template, and later use the same template for a new payment. This flow is similar to omni payment widget, except template selector shows templates of more than one payment type. User is able to see templates for all payment types configured for wizard and based upon given permissions.

    • Ability to create a payment from Manage Templates widget. In the Manage Template widget, list and detail views allow the user to create a payment from a specific template. The user is navigated to the payments form with all the details pre-filled from the saved template.

    • Delivery option and Fee sharing option in a template are shown preselected to the user, when such template is used for a payment.

    • Extension slots of Manage payments are also added to Manage payment templates.

  • Web: Ability to configure first day of week in date selector

    Date selector is added with ability to configure any particular day as the first day of week. Structure of the month shown in date selector starts with the configured day as first day in its layout.

Bug fixes

This release includes the following bug fixes:

  • MAINT-9964: BeneficiarySelectorComponent only displays the first account of the contact

    If a contact has more than one accounts, currently only first contact account is shown in the beneficiary selector. Now it is possible to see all of the contact accounts in the beneficiary selector. Additionally it shows name of both contact and account for each of these accounts.

  • MAINT-10493: Omni Payment configuration saves contacts at user level

    A contact created from beneficiary details of a payment using 'Save as a contact' is saved on User level. This affects approval flow as well as contact is not shared within Service Agreement. Both Omni payment widget and Payment wizard widget are added with a configuration enableSavePaymentAsTemplate to save a contact on a certain level. Availble levels are 'USER', 'SA' or 'LE'. Default is 'SA'.

Breaking changes

This release includes the following breaking changes:

  • Removal of 'Initiate Payment' and 'Initiate Business Payment' widget. Both Initaite payment (Deprecated in 3.8.0) and Initiate Business Payment (Deprecated in 3.7.0) are removed from Business Banking Collection. Omni payment widget is replacedment for both of these widgets, refer - https://community.backbase.com/documentation/Business-Banking/latest/web_payments#omni_payment.

Products v3.9.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: Aggregated balances in View business account list widget

    The overall balance aggregations grouped by currency are now visible in View business account list widget on top or at the bottom of the accounts list, in order for end customers to easily have access to the company’s financial position and make appropriate business decisions.

The position of the aggregations is handled by the bank user through Experience Manager, by introducing a new property.

Aggregated balances position property has the following options: - top - default option - aggregations are displayed at the top of the accounts list. - bottom - aggregations are displayed at the bottom of the accounts list.

Bug fixes

This release includes the following bug fixes:

  • PRSUM-7118: Business - Incorrect message for multiple accounts selection

    When more than one account is selected in the account selector, the error message was stating "size must be between 1 and 36", whilst is should instead provide the following message: "You have selected multiple accounts. Please select only one account to display the account information."

  • PRSUM-7137: Business - show error for multi account selection

    When apllying multi-select for the account selector, opening up the balance history and selecting several accounts, the balance history chart displays the first account with the incorrect error message of "not entitled to see this arrangement. An error occurred when loading balance history table. Try reloading the page or contacting the system administrator." This now says "You have selected multiple accounts. Please select only one account to display the account information" .

User Self Service v3.9.0

This release note explains changes to this capability.

Bug fixes

This release includes the following bug fix:

  • MAINT-10678: Double submission of change password form on submit

    The Manage user security widget submitted the change password request twice when a form was filled in and the Submit button was clicked. To resolve this and address accessibility concerns, the mousedown event that caused this issue has been removed from the Submit and Cancel buttons.

    The mousedown event was originally added to stop the validation from modifying the UI while submission occurred. This validation has been moved to after the submission event.

Migration

When you migrate from one version of this product to another, best practice is to update incrementally through all minor and major versions in between.

Follow these steps to update an existing deployment:

  1. Update non-prefixed references in the data-ang project

    To move from the deprecated non-prefixed ApiModule, BASE_PATH, and ConfigurationParameters, follow these steps:

    1. Check the public_api.ts file in data-ang project for the correct prefix to use. For example, the Manage user security widget requires the prefix User.

    2. Update your project to use the prefix. For example, replace ApiModule, BASE_PATH, and ConfigurationParameters with UserAPIModule, USER_BASE_PATH, and UserConfigurationParameters.
