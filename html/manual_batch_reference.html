Create manual batch payments

The Create manual batch payments widget expects a configuration defining the available batch types, the steps that make up the wizard, and the form fields in the steps.

The overall structure of the configuration is as follows:

  • Main configuration

    • Batch type configurations: The batch types this configuration supports

      • Form group configurations: A group of form controls

        The form group is linked to a step in the wizard.

        • Form field configurations: A form control

    • Step configurations: The steps in the wizard

Main configuration

The TypeScript interface of the main widget configuration is as follows:

export interface ManualBatchesConfig {
  availableTypes: Array<ManualBatchTypeConfig>;
  enableTypeSelection: boolean | Array<StepName>;
  steps?: Array<StepperConfig>;
  options?: MultiPaymentWidgetOptions;
}
Name Description

availableTypes

The list of Batch type configurations available in the Create manual batch payments widget.

enableTypeSelection

Flag to show batch type selection component. It accepts a collection of step identifiers where the component will be shown.

steps

Optional. List of Step configurations. Each step configuration defines a step in the wizard.

options

Optional. Configuration options.

Batch type configuration

The batch type configuration defines the name, the form groups, and the associated business function. The TypeScript interface is as follows:

export interface ManualBatchTypeConfig {
    name: string;
    paymentType: string;
    fields: Array<PaymentFormFieldConfig | PaymentFormGroup>;
    customFields?: PaymentComponentsMap;
    resourceName?: string;
    businessFunction?: string;
    hooks?: PaymentHooksOptions;
    options?: PaymentTypeOptions;
}

This configuration has the following properties:

Name Description

name

Name of the batch type.

paymentType

The payment type that is sent to DBS when the all the payments are created. For example: INTRACOMPANY_PAYMENT_CREDIT_USA.

fields

List of Form group configurations available in this batch type. In the form group configuration, you can map the form group to one of the steps of the wizard.

resourceName

Optional. Access Control resource that the customer needs to access this batch type.

businessFunction

Optional. Access Control business function that the customer needs to access this batch type.

customFields

Optional. Map of custom form components for new payment components or to override the existing components.

hooks

Optional. Map of functions that support extending the functionality of the batch type configuration. See Provide hooks to the manual batch configuration.

options

Optional.

Out-of-the-box batch type configurations

The following batch type configuration is supported out of the box:

Batch type Exported name Required business function

Intracompany payment credit USA

INTRACOMPANY_PAYMENT_CREDIT_USA

BATCH_INTERNAL_TRANSFERS

Batch - Intracompany Payments

Step configuration

The wizard can consist of multiple steps that show one or more form groups. Each step has the following configuration:

export interface StepperConfig {
  stepName: StepName;
  stepDisplayValue: string;
  isActive: boolean;
  isChecked: boolean;
  buttonClass: Array<string>;
  buttonConfig: Array<StepperButtonConfig>;
  cssClasses?: Array<string>;
  footerDetail?: MultiPaymentFooterOptions;
}

This configuration has the following properties:

Name Description

stepName

Name of the step. Used as identifier in Form group configuration stepName.

stepDisplayValue

Label to be shown in the bb-stepper-step-ui component.

isActive

A flag indicating if the step should be shown initially as active.

isChecked

A flag indicating if the step should be shown initially as checked.

buttonClass

An array of CSS classes to style the button bar of the step. For example ['fixed-bottom', 'bb-button-bar—​sticky'] to fix the button bar to the bottom and make it sticky.

buttonConfig
Untitled

The collection of buttons configuration for this step.

export interface StepperButtonConfig {
  type: string;
  text: string;
  action: StepperAction;
  hooks?: MultiPaymentHooksConfig;
}

cssClasses

An array of CSS classes to style the step container. For example: ['card', 'card-body'] to have a card style step.

footerDetail
Untitled

The configuration to show details in the footer alongside with the buttons

export interface MultiPaymentFooterOptions {
  totalCount: boolean;
  totalAmount: boolean;
}

Form group configuration

The form group configuration is used to represent a form group fields together. It is also possible to create nested form groups.

export interface MultipleFormGroup {
    name: string;
    title?: string;
    stepName?: string;
    collapsible?: boolean;
    fields: Array<PaymentFormFieldConfig | MultipleFormGroup>;
    cssClasses?: Array<string>;
}

This configuration has the following properties:

Name Description

name

Name of the form group. This is the Angular form group name, so it should be a string with no spaces.

title

Optional. Title for the form group.

stepName

Optional. The step in the wizard where the form group is visible.

collapsible

Whether users can collapse and expand the form group. If enabled, also set a title for this form group, in order to display a heading in the collapsed state.

fields

Array of Form group configurations or Form field configurations.

cssClasses

An array of CSS classes to style the group.

Form field configuration

The form field configuration defines a single form control. It is not possible to nest form fields.

export interface PaymentFormFieldConfig {
    name: string;
    type: PaymentComponents | string;
    options: PaymentFormFieldOptions;
    hooks?: PaymentFormFieldHooksOptions;
}

This configuration has the following properties:

Name Description

name

Name of the form control. This is the Angular form control name, so it should be a string with no spaces.

type
Untitled

The type of form control, a string that maps to a form control component name. See Components below for more details.

export const PaymentComponentsMap = {
    text: PayordTextboxComponent,
    textarea: PayordTextAreaComponent,
    address: PayordAddressComponent,
    select: PayordSelectComponent,
    radio: PayordRadioComponent,
    debitAccountSelector: PayordDebitAccountSelectorComponent,
    beneficiarySelector: PayordBeneficiarySelectorComponent,
    schedule: PayordScheduleComponent,
    amount: PayordAmountComponent,
    checkbox: PayordCheckboxComponent,
    switch: PayordSwitchComponent,
    date: PayordDateComponent,
    number: PayordNumberComponent,
    phone: PayordPhoneComponent,
    internalAccountSelector: PayordInternalAccountSelectorComponent,
    compactAmount: PayordCompactAmountComponent,
    scheduleHeader: PayordScheduleHeaderComponent,
};

options

A set of properties to customize each form control, see Generic form field options.

hooks

Optional. Map of functions that support extending the functionality of the form control. See Provide hooks to the form field configuration.

Generic form field options

The below options are common to all form components. Any additional option specific to a particular field will be specified in the components section.

Option name Description

label

The label for the form control. Defaults to an empty string.

placeholder

The placeholder for the form control. Defaults to an empty string. The placeholder option is not applicable to checkbox, radio and switch components.

cssClasses

An array of CSS classes to style the form field. For example: ['d-inline-block', 'col-12', 'col-md-6']. The classes in the form field config should occupy 50% or less of the width of the form in medium and large screens.

validators

An array of angular validation functions that the widget applies to the form control. For example: [Validators.required, routingTransitNumberValidator].

asyncValidators

Similar to validators an array of asynchronous validators that the widget applies to the form control.

validationMessages

An array of custom validation messages for different error types. For example: [{ name: 'required', message: 'End date is required' }]. See Provide custom validation messages to the form field.

defaultValue

It is the initial value the form control.

dependants

An array of names of form fields which are depending on the current field. The field names provided in dependants are hidden by default and are displayed based on a value or condition. See Toggle fields.

activateDependantsOn

A value or a function that determines when the dependants should be visible. See Toggle fields.

mapToField

This property helps to map the correct parent of the form field. When a form control is placed in a group or section, it must be mapped to the appropriate parent so that the field value is sent correctly to the backend. See Reposition fields.

isConditionalMandatory

This property helps to specify dependant fields as mandatory when they are visible. Don’t provide the dependant fields with Validator.required, since they are hidden by default. In those cases, set isConditionalMandatory to make the field required only when it is visible.

helperText

Optional. Text that is visible next to form control’s label.

description

Optional. Description of the form control. It is visible below the form control.

Components

This section lists the out-of-the-box supported payment form components and ways to configure them.

  • Text component: This component is similar to the bb-input-text-ui from Web Design System. It accepts the input properties of bb-input-text-ui as options. The textbox component accepts an option showCharCounter which shows the current count of typed characters.

    {
    type: PaymentComponents.text,
    name: 'name',
    options: {
        validators: [Validators.required, noWhitespaceValidator as ValidatorFn],
        validationMessages: [{ message: 'Batch name is required', name: 'required' }],
        label: 'Batch name',
        showCharCounter: true,
        maxLength: 25,
        cssClasses: ['w-50'],
    },
},

  • Textarea component: This component is similar to the bb-input-textarea-ui from Web Design System. It accepts the input properties of bb-input-textarea-ui as options. Provide collapseOptions to make the text area collapsed by default.

    • Regular textarea:

      {
    name: CounterPartyFields.remittanceInformation,
    type: PaymentComponents.textarea,
    options: {
        label: 'Message to beneficiary (optional)',
        placeholder: 'Type your message here',
        showCharCounter: true,
        minLength: 0,
        maxLength: 140,
        rows: 3,
        cols: 50,
        cssClasses: ['d-inline-block', 'col-12', 'px-0'],
        mapToField: PaymentBaseFields.counterparty,
    },
},

    • Collapsed textarea:

      {
    name: CounterPartyFields.remittanceInformation,
    type: PaymentComponents.textarea,
    options: {
        label: 'Message to beneficiary (optional)',
        placeholder: 'Type your message here',
        showCharCounter: true,
        minLength: 0,
        maxLength: 140,
        rows: 3,
        cols: 50,
        cssClasses: ['d-inline-block', 'col-12', 'px-0'],
        mapToField: PaymentBaseFields.counterparty,
        collapseOptions: {
            btnColor: 'link',
            btnSize: 'md',
            hideIcon: false,
            icon: 'add',
            iconSize: 'md',
            label: 'Add message to beneficary',
        }
    },
},

  • Select component: This component is similar to the bb-dropdown-single-select-ui from Web Design System. It accepts the input properties of bb-dropdown-single-select-ui as options.

    {
    name: CounterPartyFields.accountType,
    type: PaymentComponents.select,
    options: {
        label: 'Account type',
        validators: [Validators.required],
        options: [{
            label: 'Savings',
            value: 'SAVINGS',
        }, {
            label: 'Checking',
            value: 'CHECKING',
        }],
        placeholder: 'Select an account type',
        cssClasses: ['d-inline-block', 'col-6', 'align-top', 'pr-0'],
    },
},

  • Radio component: This component is similar to the bb-input-radio-group-ui from Web Design System. It accepts the following input options.

    {
    name: CounterPartyFields.standardEntryClass,
    type: PaymentComponents.radio,
    options: {
    label: 'Standard entry class (SEC Type)',
    validators: [Validators.required],
    cssClasses: ['d-inline-block', 'col-6', 'align-top', 'pl-0'],
    mapToField: PaymentBaseFields.counterparty,
    horizontal:true,
    preselect: true,
    options: [{
            label: 'Personal(PPD)',
            value: 'PPD',
        },{
            label: 'Business(CCD)',
            value: 'CCD',
        },
    ]},
},

  • Checkbox component: This component is similar to the bb-input-checkbox-ui from Web Design System. It accepts the following input options.

    {
    type: PaymentComponents.checkbox,
    name: RemittanceInfoFields.urgent,
    options: {
        label: 'This payment is urgent',
    },
},

  • Switch component: This component is similar to the bb-switch-ui from Web Design System. It accepts the following input options:

    {
    name: 'isCorrespondentBankTransfer',
    type: PaymentComponents.switch,
    options: {
        label: 'Transfer through correspondent bank',
        cssClasses: ['d-block', 'align-top', 'pl-0', 'mb-0'],
        labelPosition: 'right',
    },
},

  • Number component: This component is similar to the bb-input-number-ui from Web Design System. It accepts the following input options:

    {
  name: ScheduleFields.repeat,
  type: PaymentComponents.number,
  options: {
    isConditionalMandatory: true,
    label: 'Repeat',
    min: 2,
    max: 200,
    defaultValue: 2,
    validationMessages: [
      { name: 'required', message: 'Value is required' },
      { name: 'max', message: 'Value exceeds maximum allowed occurances' },
    ],
  },
},

  • Phone component: This component is similar to the bb-input-phone-ui from Web Design System. It accepts the following input options:

    {
  name: 'phoneNumber',
  type: PaymentComponents.phone,
  options: {
    label: 'Beneficiary phone number',
  },
},

  • Amount component: This component is similar to the bb-amount-ui from Web Design System. The default value for the amount should be an object similar to the given example. It accepts the following input options:

    {
    type: PaymentComponents.amount,
    name: RemittanceInfoFields.amountCurrencyGroup,
    options: {
        label: 'Amount',
        cssClasses: ['col-8', 'pl-0', 'mb-0'],
        currencies: ['USD'],
        defaultValue: {
          amount: '5.00',
          currency: 'USD',
        },
    },
},

  • Compact amount component: This component is a single combined field and not split into currency, amount, and decimal. It accepts the following input:

    {
  type: PaymentComponents.compactAmount,
  name: RemittanceInfoFields.amountCurrencyGroup,
  options: {
    label: 'Amount',
    cssClasses: ['bb-amount-input__field--large', 'bb-block', 'bb-block--lg', 'col-md-6'],
    currency: 'USD',
    autoDecimal: true,
    validationMessages: [
      {name: 'invalidAmount',
      message: 'Please add amount of this transfer',
      },
    ],
  },
},

    autoDecimal: Sets the decimal automatically on the UI when the bank customer starts typing in the input field. currency is used as initial currency.

  • Date component: This component is similar to the bb-input-datepicker-ui from Web Design System. It accepts the following input options:

    {
    type: PaymentComponents.date,
    name: 'requestedExecutionDate',
    options: {
        validators: [pastDateValidator as ValidatorFn, Validators.required],
        validationMessages: [{ message: 'Execution date cannot be in past', name: 'pastDate' }],
        label: 'Execution date',
        cssClasses: ['w-50'],
        overrideDateFormat: 'dd/MM/yyyy',
        placeholder: 'DD/MM/YYYY',
    },
}

  • Credit debit selector component: This component allows customers to switch from credit to debit for the same batch type. For now only credit is supported.

    A sample config for credit debit selector is below:

    {
  type: PaymentComponents.creditDebitSelector,
  name: 'creditDebitIndicator',
  options: {
    label: 'Internal transfers',
    choices: [
      { type: 'CREDIT', name: 'Send', description: 'Send money to multiple own accounts' },
    ],
  },
}

  • Multipayment account selector component: This component is similar to the bb-account-selector-ui component from Web Design System. The widget injects the debit accounts as property named items$ in options at runtime. When overriding PaymentComponents.debitAccountSelector using customComponents (see Override existing form components), retrieve the debit accounts by subscribing to the items$ observable that is injected into config options. The type of items$ is as follows:

    import { InternalAccountFormItem } from '@backbase/payments-common-ang';
items$: Observable<Array<InternalAccountFormItem>>;

    A sample config for debit account selector is below:

    {
  type: PaymentComponents.multiPaymentAccountSelector,
  name: InitiatorFields.initiatorAccountGroup,
  options: {
    label: 'From',
    placeholder: 'Select an account',
    validators: [Validators.required],
    validationMessages: [{ message: 'Account is required', name: 'required' }],
    arrangements: {
      privilege: 'create',
      ignoredProductKindNames: [ProductKinds.creditCard, ProductKinds.debitCard],
    },
    cssClasses: ['w-75'],
    filterItems: false,
    required: true,
    showBankBranchCode: true,
  },
}

  • Multi payment card component: This component renders payments on demand. It allows customers to create, edit, or remove payments. It can handle up to a 1,000 payments thanks to the virtual scroll technique, that only renders the visible payments while scrolling up and down. It takes the maximum number of payments to manage from the configuration options, it is not allowed to create more payments than that maximum. The arrangements request can also be configured, in options.arrangements, to filter some arrangement types and which privilege is used to make the request.

    import { InternalAccountFormItem } from '@backbase/payments-common-ang';
items$: Observable<Array<InternalAccountFormItem>>;

    A sample config for debit account selector is below:

    {
  type: PaymentComponents.multiPaymentCard,
  name: InitiatorFields.initiatorAccountGroup,
  options: {
    label: 'To',
    helperText: 'You can add up to %s transfers',
    validators: [minimumOneAccount as ValidatorFn],
    maximumNumberOfPayments: 100,
    validationMessages: [{ message: 'Minimum one transfer is required', name: 'empty' }],
    arrangements: {
      privilege: 'view',
      ignoredProductKindNames: [ProductKinds.creditCard, ProductKinds.debitCard, ProductKinds.loanAccount],
    },
    fieldsConfig: {
      counterpartyName: {
        validators: [Validators.required, accountsValidation(PaymentBaseFields.initiator)],
        placeholder: 'Enter or select account',
        dropdownIcon: '',
        ariaLabel: '',
        validationMessages: [
          { message: 'Account is required', name: 'required' },
          { message: 'Account is same as from', name: 'sameAccount' },
        ],
      },
      amount: {
        validators: [Validators.required, amountGreaterThanZero as ValidatorFn],
        placeholder: '0.00',
        ariaLabel: '',
        currency: 'USD',
        maxLength: 10,
        validationMessages: [
          { message: 'Amount is required', name: 'required' },
          { message: 'Amount is required', name: 'invalidAmount' },
        ],
      },
      description: {
        validators: [],
        placeholder: 'Enter Description',
        label: 'Add Description',
        helperText: 'Optional',
        title: 'Description',
      },
    },
    hooks: {
      onBlur: verifyOnBlur,
    },
  },
}
