All capabilities v3.7.0

This release note lists the changes to all capabilities in this release.

Announcements

This release includes the following announcements:

  • Renamed widget collection

    Widget Collection Business Banking is now named Business Banking Widget Collection. This name change is only for the documentation, and does not have any impact on the content of the product.

Deprecations

This release includes the following deprecations:

  • In 3.7.1:

    • Support ceases for the triangular services for a number of backend capabilities. The widgets in this collection are updated to use the new lean services.

      The following Digital Banking Services capabilities have implemented the lean architecture this release:

      • Access Control

      • Account Statements

      • Audit

      • Bill Pay

      • Personal Finance Management

      • Products

      • Transactions

      • Users

      The following Entitlements Services capabilities have implemented the lean architecture this release:

      • Access Control

      • Limits

      • Users

      To see how to roll back the lean architecture and use the deprecated triangular architecture, see Change service type.

  • In 3.8.0:

    • Support ceases for Angular 8. All Angular widgets, journeys, and libraries will upgrade to Angular 10.

  • In the version released in January 2021:

    • Support ceases for Microsoft Internet Explorer 11

      Note
      		 The date for dropping support for Microsoft Internet Explorer 11 has changed. Support will now continue to January 2022.

Contacts v3.7.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Improved error details

    More context is given to errors during creating/editing contacts. For this, a complete error response is returned along with error message.

  • Possibility to search a contact with special characters like '+' in search criteria

    Now it is possible to search contacts using special characters like '+' in search criteria. All matching contacts are retrieved, which include searched special character in it along with other search criteria. This needs to be opted in by enabling the feature flag EXTRA_ENCODE_URI_PARAM￼to True in main application module.

    Starting from foundation-ang@5.0.0, @backbase/ui-ang will no longer be a part of the collection. Payments v3.7.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Possibility to filter payments created only by logged in user

    Now it is possible for users to also filter payments created by the logged in user. This possibality of filtering is added for both One-off and Standing payment orders seperately.

  • Possibilty to view Standing payment orders seperately

    Now it is possible to view Standing payment orders seperately from One-off payment orders. Now there are two seperate views/tabs, one each for One-off payment orders and Standing payment orders. Standing payment orders view also shows frequency as extra detail. Also payment details of a standing payment orders also show Start Date (mandatory), Frequency (mandatory), End Date (only if present), Ends after x occurrences (only if present), Next Execution Date (only if present).

  • Show payment details dynamically

    Now it is possible to show details of a custom payment type dynamically without customizing the Manage Payment Widget. Widget provides ootb one opinionated mapping for property labels to view the details of a payment for a custom payment type. Additionally, certain payment type dependent property labels (Payment type label, Bank branch code label) are derived from custom payment type configuration.

  • Enhancements in Omni Payment widget

    New enhancements are added in Omni Payment Widget, which includes capabilities;

    • Fix support for static translations

    • Fix custom label support for review/confirmation screens

    • Configure custom error messages for fields

    • Configure buttons differently

    • Configure individual fields in Schedule section

    • Apply CSS classes to individual form groups, instead of the complete widget

    • Click to reveal feature for textarea

    • Configure helper text, description and default value for form field components

    • Customise button section in review screen

    • Ability to provide options to omni payment widget (enable/disable approvals, specify default scheme, enable/disable templates )

    • Addition of new components - date, number , phone

  • Save a payment as a template after submitting it

    Now it is possible to save a payment as a template after submitting it. Payment details such as; Payment Type, Originator, Originator Account, Instruction Priority, Entry Class and Transfer Transaction Information are saved with the template.

  • Search, select and apply a payment template

    Now it is possible to Search, Select and Apply a payment template while creating a Payment. Desired payment template can be selected from the list of applicable payment templates in a dropdown on a payment form.

  • BETA: New omni payment configs for CHAPS, Faster Payments and SEPA

    New configs for omni payment widget are available for CHAPS, Faster Payments and SEPA payments. This is a beta (sample code) release. This feature may, unless otherwise authorized by Backbase, be used for testing purposes only and is provided on an "AS IS" basis without support of any kind. You should not use this feature in a production environment.

Bug fixes

This release includes the following bug fixes:

  • MAINT-9875: Submit and Approve button always shown in the review screen

    Omni-payment widget displays the button Submit and a dropdown with the Submit and Approve button, but the latter should be displayed only when the user has permission to approve the payment. To solve this, a new ability to provide options to the widget is added, while providing flag enableApprovals as false. These options can be provided as an input to the omni payment widget or using the withConfig method

Deprecations

This release includes the following deprecations:

  • In v3.8.0:

    • Initiate Business payment widget is deprecated in this release. It will be removed in 3.8.0.

Breaking changes

This release includes the following breaking changes:

  • Removal of ACH credit widget:

    In this release, ACH credit widget is removed. It is replaced ACH config powered omni payment widget.

  • Button section in review screen moved inside 'reviewTemplate':

    In omni payment widget the button section in review screen is moved inside the 'reviewTemplate'. If review screen of omni payment widget has been customised in a project using 'reviewTemplate', then this custom 'reviewTemplate' needs to include the button section inside the template.

  • ui-ang split out of foundation-ang:

    Starting from foundation-ang@5.0.0, @backbase/ui-ang will no longer be a part of the collection. It would be required to add @backbase/ui-ang as a separate dependency for projects and change imports containing the path @backbase/foundation-ang/ui to @backbase/ui-ang. Check foundation-ang release notes for updating using schematics and more details.

Transactions v3.7.0

This release note explains changes to this capability.

New features

This release provides the following new features:

  • Web: Ability to view the merchant name and icon of a transaction

    The View transactions table widget now provides the ability to display the merchant icon and name of a transaction so that bank customers can easily identify their transactions.

  • Web: Ability to have a more comprehensive visual indication of transactions with a new set of icons

    The View transactions table widget now provides the ability to display different icons for transactions based on available data. In addition to category icons, now transactions can display merchant icons, transaction type icons, incoming transaction icon and outgoing transaction icon. Which icon to be displayed depends on the following priority order: merchant icons > category icons > transaction type icons > Incoming and outgoing transaction icons. For example, if the merchant information is not present for a transaction, then category information is checked and an icon for category is shown, if it exists. If category information is also not present different icons are shown based on the transaction type. If transaction type information is not present, then either an incoming or outgoing transaction icon is shown.

  • Web: A refreshed transaction details view with a cleaner user interface

    View transaction details* now has a new look with a header which includes vital transaction information followed by more readable fields.

Deprecations

This release includes the following deprecations:

  • In v3.8.0:

    • TransactionCategoryIconComponent will no longer use large input in favor of a new input called size that can receive the string large and small to define the size of the icon.

    • WidgetPropertiesService will no longer make use of the getter gmapsAPIKey in favor of googleAPIKey input.

Breaking changes

This release includes the following breaking changes:

  • Web:

    • The defaultExportConfig input in BaseTransactionsControls is removed in favor of exportConfig input.

    • The transactionsList input in TransactionsNumberOfResultsFoundComponent is removed in favor of numberOfResultsFound input. The component now receives only the amount of results found.

    • Notification types NotificationMessage and NotificationModifier in TransactionsCategoriesComponent are no longer used.

    • ViewChild in TransactionsCategoriesComponent (notificationUpdate, notificationErrorUpdate) is no longer used.

    • Method pushNotification() in TransactionsCategoriesComponent is no longer used.

    • Output cancelCategory in TransactionsCategoriesComponent replaced with categoryChange.

    • Method cancelChangeCategory() in TransactionsCategoriesComponent is replaced with cancelSaveCategory().

    • Method saveCategory() in TransactionsCategoriesComponent is replaced with changeCategory().

    • The getter unsignedAmount in TransactionsTableDetailsComponent is removed. The transaction object now has a property called transactionAmountCurrency that in turn has unsignedAmount value.

    • The transaction model went through the following changes:

      • Removed productId property.

      • Removed amount property in favor of transactionAmountCurrency.amount.

      • Removed currency property in favor of transactionAmountCurrency.currencyCode.

      • Removed scheduledDate property.

      • Removed instructedAmount property in favor of instructedAmountCurrency.amount.

      • Removed instructedCurrency property in favor of instructedAmountCurrency.currencyCode.

      • Removed accountNumber property.

      • Removed debitCreditSign property.
