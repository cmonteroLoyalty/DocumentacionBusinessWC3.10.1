What is Business Banking Widget Collection

Business Banking Widget Collection provides a set of web widgets and beta (sample code) native mobile screens and journeys that cover functionality specific to the business banking segment.

With Business Banking Widget Collection, bank customers change between legal entities and service agreements, make single and batch payments, review transactions, and perform other corporate financial operations.

You import this collection into Experience Manager to build a new experience tailored to a specific financial institution.

Business Banking Widget Collection includes solutions for the following channels:

  • Web

  • Native mobile - beta (sample code)

Note
 To use all functionality in this collection, you must have deployed and configured Digital Banking Services. If not, you need to use the capability specifications and create backend services that comply with the requests for Digital Banking Services.

Download the specifications from Module/Endpoint List.

You use the widgets, screens, and journeys from this collection out of the box, or you modify them to fit a specific use case or business flow. There are several different modification methods available.

Themes

With themes, frontend developers apply visual changes to the widgets. For example, change colors, borders, fonts, or spacing.

Widget preferences

With preferences, frontend developers configure certain features of a widget through Experience Manager. For example, set the number of payment batches to be shown per page or configure payment card types. Each widget provides a different set of preferences.

Model extensions

With model extensions, frontend developers introduce additional properties in the widget. For example, if you require an additional field in the Contact form of the Contact Manager Business widget, this can be achieved by using the model extension mechanism. This mechanism can be combined with extension slots to make the fields available in the view of the widget.

Native mobile - beta (sample code)

The native mobile part of the Business Banking Widget Collection is supplied as out-of-the-box mobile screens and journeys that are configurable and themeable.

Screen

  • For Android, each screen is built as a Fragment to ensure your app is a Single-Activity Application. The main advantage of the Single-Activity Application architecture is that transitions between screens become much smoother, since an Activity is always tied to a single process. Launching a new screen within an Activity does not require starting up a new process, which improves user experience. The architecture used is Model–view–viewmodel (MVVM) with LiveData, as the most popular architecture for Android at the moment.

  • For iOS, each screen is built as a separate ViewController to ensure that it is independent. The architecture used is VIP (Clean Swift), which is Clean Architecture used in iOS and Mac projects.

The use of native components ensures that the navigation and data transfer are done in an industry-standard way. For Android, you pass on a bundle with the information that the screen requires, and for iOS you pass on the data in the constructor or initializer.

There are no restrictions on what architecture you can use within a Fragment or ViewController when building custom screens. For example, it can be Model–view–viewmodel (MVVM), Model–view–presenter (MVP), or Model–view–controller (MVC).

To ensure upgradability and simplify the API, screens do not support extension. Each screen has one or more entry points, which are ways to initialise the screen, and one or more exit points. An example of an exit point for the Transaction List screen is the action when a transaction is selected.

A screen has properties that you configure, for example:

  • Fields

  • List header type

  • Page header type

  • Theme

Journey

A journey consists of screens sharing the same user story, connected together with default navigation between them.

A journey consists of multiple screens connected in a NavGraph, which is part of the Android Navigation Component. If NavGraph is not desirable, you can also construct a journey by having one hosting fragment that contains all the child screens and can navigate between them. This approach is called a fragment with self-contained navigation. The advantage of using the Android Navigation Component is that the back stack management is handled for you. The transition between screens is often customisable, and a journey can support multiple ways of transitioning. Examples of transitions include shared element transition, slide in and out, and pop open and close. The purpose of a journey is to offer a strong and well user tested experience with flexible configuration and theming, to ensure it fits the projects' needs.

Communication

The communication between the screens is completely platform-native. If you are using the Android Navigation Component, the bundle with data is passed to the NavController’s navigate method. If you are using the fragment with self-contained navigation approach, pass it on when switching fragments on the hosting fragment.

Configuration

The configuration of screens and journeys happens through an injected data class that is provided through Koin. Koin is an easy-to-use service locator which works much like a dependency injection framework and is widely used. The reason for using a service locator is to provide ways to configure screens and journeys, without having to open them up for extending. The data class is easy to create because it is provided with a type-safe builder using DSL. For implementation projects, these techniques do not have to be used in creating custom build screens and journeys. However, we recommend doing so to ensure the app has a way of being configured.

In some cases, OOTB journeys may not cover the full set of requirements for a project. In those cases, you can construct custom user journeys and reuse screens that we offer in our capabilities. To ensure secure connectivity in this case, mobile clients still have to be used to retrieve data from DBS.
